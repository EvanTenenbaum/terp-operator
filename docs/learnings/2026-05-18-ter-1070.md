# TER-1070 Learnings (2026-05-18)

Non-obvious patterns, gotchas, and constraints discovered during the parity-frontend implementation. Captured for future agents working in this codebase.

## 1. Parity script literal-string regex (high value, gotcha)

**Type:** gotcha
**Applies to:** every new tRPC command surface in client code

`scripts/check-backend-frontend-parity.mjs` uses two different matchers:

| Surface | Matcher | Bypass example |
|---------|---------|----------------|
| Command | regex `runCommand\(\s*['"\`]${name}['"\`]` | `runCommand(mode === 'create' ? 'A' : 'B', ...)` matches NEITHER A nor B |
| Query | substring `clientText.includes('queries.${name}')` | Lenient — any reference passes, including type annotations |

**Rule:** Every `runCommand(...)` call MUST use a literal string. If a component needs to dispatch one of several commands, use separate `runCommand` invocations or `switch`/`if` with literal strings in each branch — never a ternary or computed name.

**Why:** The parity audit blocks PRs and gives a misleading "missing surface" error when this rule is violated. Caught during plan-review-gate iteration 1.

## 2. tRPC `createCaller` test pattern (high value, pattern)

**Type:** pattern
**Applies to:** `src/server/services/*.test.ts` files that test tRPC procedures

The `TrpcContext` in this repo (`src/server/trpc.ts`) is `{ req, res, io, user }` — NOT `{ session: { user } }`. `protectedProcedure` reads `ctx.user` directly and throws UNAUTHORIZED if it's null.

**Pattern that works in both standalone and full-suite runs:**

```typescript
import { describe, expect, it, vi, beforeEach } from 'vitest';

let mockExecute: ReturnType<typeof vi.fn>;

const mockCtx = {
  req: {} as any,
  res: {} as any,
  io: {} as any,
  user: { id: 'test-user-id', name: 'Test', email: 'test@test.com', role: 'manager' as const }
};

describe('myQuery', () => {
  beforeEach(() => {
    // CRITICAL: reset modules + doMock per test, or a previous test file's
    // import of '../db' caches the real module and our mock never applies.
    vi.resetModules();
    mockExecute = vi.fn();
    vi.doMock('../db', () => ({
      db: { execute: mockExecute },
      pool: { query: vi.fn(), on: vi.fn(), connect: vi.fn() },
      pingDatabase: vi.fn(async () => true)
    }));
  });

  it('...', async () => {
    mockExecute.mockResolvedValueOnce({ rows: [...] });
    const { queriesRouter } = await import('../routers/queries');
    const caller = queriesRouter.createCaller(mockCtx);
    // ...
  });
});
```

**Why:** Top-level `vi.mock('../db', ...)` works in standalone runs but fails when other tests import `../db` first and cache the real module. Also, `'../db'` exports `db`, `pool`, and `pingDatabase` — partial mocks of just `db` cause the queries router to fail loading because `pool` (used by session middleware/auth) is undefined.

## 3. v8 coverage instrumentation timeouts (medium value, gotcha)

**Type:** gotcha
**Applies to:** `vitest.config.ts` and any test file with userEvent or dynamic imports

When `@vitest/coverage-v8` is enabled, tests run 3-4× slower due to instrumentation. The default 5s `testTimeout` causes user-event and dynamic-import tests to timeout under coverage even though they complete in <2s standalone.

**Fix:** Set `testTimeout: 30000` in `vitest.config.ts` `test:` block. Without coverage the slack is unused; with coverage it prevents flakes.

## 4. Built server runs stale code until restarted (medium value, gotcha)

**Type:** gotcha
**Applies to:** any local QA after `pnpm build`

`pnpm build` writes new bundles to `dist/`, but a running `node dist/server/index.js` process keeps the previously-loaded modules in memory. Code changes from the build don't take effect until the process is killed and restarted.

**Fix when QA'ing locally:** `pkill -f "node dist/server/index.js" && sleep 2 && NODE_ENV=... node dist/server/index.js &`

## 5. `secure: isProd` cookie blocks HTTP local QA (high value, gotcha)

**Type:** gotcha
**Applies to:** `src/server/auth.ts` session config; any local prod-mode testing

The session cookie sets `secure: isProd`. When you run `dist/server/index.js` with `NODE_ENV=production` on plain HTTP `localhost`, the browser silently drops the cookie (Set-Cookie present, but not stored), and every authenticated request returns 401. Login API call returns 200 but `document.cookie` is empty.

**Workarounds for local QA only (never for real deployments):**
- Run server with `NODE_ENV=development` (loses prod static-file serving — server tries to redirect to APP_ORIGIN/Vite dev server)
- OR temporarily patch the built file: `sed -i.qa-bak 's/secure: isProd,/secure: false,/' dist/server/index.js`
- OR set up local HTTPS via Caddy/mkcert (proper but more setup)

## 6. CSP `upgrade-insecure-requests` traps the browser (medium value, gotcha)

**Type:** gotcha
**Applies to:** Playwright-driven QA against `localhost`

`app.ts` uses helmet's default CSP which includes `upgrade-insecure-requests`. Once the browser sees this directive for a host, subsequent HTTP navigation to that host can fail with `ERR_CONNECTION_REFUSED` (HTTPS upgrade attempt against an HTTP-only server). Closing the page doesn't always clear the in-memory cache.

**Workaround:** Navigate to `http://127.0.0.1:8787` instead of `http://localhost:8787` — different host, different CSP state.

## 7. Grid SQL must select every column the UI expects (high value, pattern)

**Type:** pattern
**Applies to:** `src/server/routers/queries.ts` `gridSql()` functions

ag-Grid renders columns by `field: 'name'` — if the SQL doesn't include that key, the cell silently renders blank. The inline `onCellCommit` handler may still WORK (it sends the field via a command that writes to the DB column), but the value never displays because the next grid fetch doesn't return it.

**Symptom found this session:** `purchaseOrders` grid had a `prepaymentAmount` column definition + inline edit handler, but the SELECT omitted `po.prepayment_amount`. The column had always been blank. Fixed in commit `723fcc8` by adding `po.prepayment_amount as "prepaymentAmount"` to the SELECT.

**Rule when adding/auditing grid views:** for every column in the corresponding `columnsByView` entry, verify the SELECT projects a matching aliased column.

## 8. `as unknown as` casts hide `notNull()` violations (high value, code_quirk)

**Type:** code_quirk
**Applies to:** command handlers in `src/server/services/commandBus.ts`

Pattern found in `recordVendorPrepayment` (and surfaced during manual QA — runtime crash, not caught by typecheck or by component tests):

```typescript
vendorBillId: null as unknown as string, // Will be linked when bill is created
```

The schema declares `vendor_bill_id: uuid().references(...).notNull()`. The TS cast bypasses the type check, but Postgres rejects the insert with a not-null violation. The command had never succeeded in any environment.

**Anti-pattern signal:** `as unknown as X` followed by a value that doesn't actually satisfy `X` is a load-bearing lie. If the field truly is optional in some lifecycle stage, change the schema to `.notNull(false)` or use a sentinel/placeholder UUID, but don't cast `null` to a non-null type.

**Detection idea:** grep for `null as unknown as` and cross-check the schema column's nullability.

## 9. Workflow note: backend reseeded between QA sessions

**Type:** code_quirk
**Applies to:** manual QA against staging DB via SSH tunnel

The staging DB sometimes gets reseeded with a much smaller dataset (~2 POs vs ~170) when the server is restarted with certain environments. Don't rely on stable IDs across sessions; query for IDs by `poNo` or other natural keys when scripting QA setup.
