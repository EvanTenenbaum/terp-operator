### 🛠️ Scripts

#### `/scripts/create-test-processor.ts`
Creates a test processor directly in the database (bypasses UI).

**Usage:**
```bash
pnpm exec tsx scripts/create-test-processor.ts
```

**Creates:**
- Name: Test-Crypto-Percentage
- Type: crypto
- Fee: 3.5% percentage
- User split: 25%
- Processor split: 75%

**When to use:**
- Setting up test environment quickly
- Unblocking QA when UI creation is broken
- Creating consistent baseline data for automated tests

**Extend it:**
To create different processor types, copy the script and modify the values:
```typescript
.values({
  name: 'Test-Check-Fixed',
  processorType: 'check',
  feeType: 'fixed',
  feePercentage: null,
  feeFixedAmount: '2.00',
  defaultUserSplit: '50',
  defaultProcessorSplit: '50',
  active: true
})
```

## QA Workflow

### For New Features

1. **Read the runbook** - Check if one exists for your feature
2. **Set up test data** - Use setup scripts to create baseline data
3. **Follow test scenarios** - Execute each scenario, document results
4. **Verify calculations** - Compare actual vs expected values
5. **Check edge cases** - Test overrides, different types, error conditions
6. **Document issues** - Report bugs with specific repro steps and expected behavior

### For Automated Testing

1. **Read navigation guide FIRST** - Understand state-based routing
2. **Use correct navigation methods** - Sidebar clicks, not URLs
3. **Wait for state changes** - Add appropriate wait conditions
4. **Verify calculations programmatically** - Don't just check for presence
5. **Take screenshots on failures** - Visual evidence helps debugging

### For Bug Reproduction

1. **Check common issues** - Runbook has known problems and fixes
2. **Verify test data** - Ensure processors/customers/etc. exist
3. **Check console** - Browser console errors often reveal root cause
4. **Test in isolation** - Reproduce with minimal steps
5. **Document environment** - Browser, data state, steps to reproduce

## Creating New Runbooks

When documenting QA for a new feature, include:

1. **Prerequisites** - What needs to be set up first
2. **Setup instructions** - How to create test data
3. **Test scenarios** - Step-by-step with expected results
4. **Calculation formulas** - For features with math/logic
5. **Common issues** - Known problems and solutions
6. **Automation notes** - Playwright examples for key interactions
7. **Success criteria** - Clear checklist of what "passing" means

**Template structure:**
```markdown
# [Feature Name] QA Runbook

## Quick Start
## Prerequisites
## Setup
## Test Scenarios
  ### Scenario N: [Name]
  **Steps:**
  **Expected Results:**
## Common Issues
## Success Criteria
## Related Files
```

## Script Development Guidelines

When creating QA setup scripts:

- **Place in `/scripts/` directory** - Not `/tmp/` (won't work due to import paths)
- **Use TypeScript** - Leverage type safety and existing schema types
- **Import from project** - Use relative imports to `/src/server/` modules
- **Return created data** - Log IDs/details for use in subsequent tests
- **Handle errors gracefully** - Exit with appropriate codes
- **Document in this README** - Add entry explaining what it does and when to use it

**Example:**
```typescript
import { db } from '../src/server/db';
import { myTable } from '../src/server/schema';

async function createTestData() {
  const record = await db.insert(myTable).values({ /* ... */ }).returning();
  console.log('Created:', record[0]);
  process.exit(0);
}

createTestData().catch(err => {
  console.error('Error:', err);
  process.exit(1);
});
```

## Known Limitations

### State-Based Routing
- Direct URL navigation doesn't work
- Must use sidebar/buttons for navigation
- Can't deep-link to specific views

**Impact:** Automated tests must navigate through UI, can't use `goto()` for views

### Rounding Discrepancies
- Financial calculations use separate rounding steps
- Final totals may vary by $0.01-$0.26

**Impact:** Test assertions need acceptable ranges, not exact matches

### Grid Data Loading
- AG Grid uses virtualization
- Not all rows may be in DOM at once
- Totals calculated via SQL, not client-side aggregation

**Impact:** May need scrolling or filtering to find specific rows in large datasets

## 🎭 Persona Flow Framework

AI-agent-executable flows for all 8 operator personas. Each flow gives an autonomous
browser agent systematic instructions to replicate a real user journey and report findings.

**Start here:** [`persona-flows/REGISTRY.md`](./persona-flows/REGISTRY.md) — master index of all 26 flows.

### Quick dispatch

```
# Run a specific flow
Load: docs/qa/persona-flows/sales-operator/01-instant-sale-normal.md
      docs/qa/persona-flows/_shared/navigation-primer.md
      docs/qa/persona-flows/_shared/seed-state-reference.md
Then: execute the flow

# Run all Critical-tier flows (required before any ship decision)
Filter REGISTRY.md by Risk = Critical — load and execute each in order
Includes: _cross-persona/01-purchase-to-payment-lifecycle.md
          _cross-persona/02-intake-reversal-mid-sale.md
          payments-accounting/03-vendor-bill-payment-lifecycle.md

# Run all flows for one persona
Load: docs/qa/persona-flows/[persona]/_persona.md + all 3 scenario files
```

### Before any run
1. `pnpm db:seed:realistic` — confirm seed state (or check `_shared/seed-state-reference.md` for current entity list)
2. App running at `http://127.0.0.1:5173`
3. Load `_shared/navigation-primer.md` + `_shared/seed-state-reference.md` alongside each scenario

Run reports go to: `runs/YYYY-MM-DD-[scope]-report.md`
Screenshots go to: `runs/screenshots/`
