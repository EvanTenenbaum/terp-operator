# Phase 3 Readiness — Money Loop

Date: 2026-05-12
Status: ready after Phase 0b; benefits from Backend Packet A

## Purpose

Make receiving money and paying money feel like fast ledger entry, not a scattered form process. Money In and Money Out must be easy to start, traceable, allocatable, and reversible.

## Required Reading

- `docs/product/work-loops.md` Loop 4
- `docs/product/capability-registry.md` CAP-004, CAP-022, CAP-024, BE-010
- `docs/design/replication-playbook.md` R1, R2, R4, R12, R15

## Prerequisites

- Payments and Vendor Payouts routes are parity-green.
- Existing payment/vendor commands pass baseline command tests.
- Role behavior for operator/manager/owner/viewer is understood before visible controls move.

## Acceptance Evidence

- Money In and Money Out are both obvious from Keel or their routes.
- Payment row validates client, method, amount, and invoice/buyer-credit state in plain language.
- Negative client amount self-labels as buyer credit/down payment.
- Allocation panel supports partial payment and overpayment without modal wizard.
- Scheduled vendor payout requires a real event date/time or explicit scheduled record.

## Smoke Test

1. Create Money In draft.
2. Allocate partial amount across open invoices.
3. Record an overpayment as buyer credit.
4. Create scheduled vendor payout.
5. Record payout.
6. Open relationship drawer and verify directional buyer/vendor balances are not netted.

## Rollback

Disable money-loop view subflag. No schema rollback unless draft persistence introduces a migration; if it does, include reverse migration and fixture proof.
