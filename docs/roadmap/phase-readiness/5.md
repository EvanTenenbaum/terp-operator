# Phase 5 Readiness — Recovery and Closeout

Date: 2026-05-12
Status: ready after Backend Packet A

## Purpose

Make mistake recovery and period closeout safe, row-origin, reversible where possible, and impossible to archive past unsafe state.

## Required Reading

- `docs/product/work-loops.md` Loop 6
- `docs/product/capability-registry.md` CAP-009, CAP-010, CAP-020, CAP-025, CAP-026, CAP-027, BE-005, BE-010
- `docs/design/replication-playbook.md` R1, R4, R10, R12, R15

## Prerequisites

- Backend Packet A complete or explicitly paired with this phase.
- `closeoutPreview` and `archivePeriod` blockers are documented and compared.
- Reversal completeness matrix exists.

## Scope

| Item | Product reason | Capability IDs | Recipe |

## Feature Flag

Use canvas flag. Archive hardening is not optional once verified; unsafe archive should not be feature-flagged off in production.

## Acceptance Evidence

- Posted row history exposes command ID and reversal preview where reversible.
- Reversal preview lists affected entities and plain-language consequence.
- Failed command can be retried or documented as terminal with reason.
- Closeout preview and archive command refuse the same unsafe blockers.
- Archive artifacts include deterministic control totals.

## Smoke Test

1. Open a posted sale row.
2. Open command history.
3. Preview reversal.
4. Cancel preview, then run a retry on a failed seeded command if fixture exists.
5. Preview closeout with unsafe rows.
6. Confirm archive action is disabled/refused until blockers clear.

## Rollback

Frontend drawer changes can be disabled with canvas flag. Backend archive hardening rollback requires owner approval because it weakens safety.
