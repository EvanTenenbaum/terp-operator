## 1. Chosen Architecture

**Approach A — shared `document_snapshots` foundation + server-side external projection allowlists.**

A single `document_snapshots` table stores the authoritative internal payload for every finalized document. Per-type pure projection modules define additive allowlists (`EXTERNAL_FIELDS`, `PROJECTION_VERSION`, `projectExternal`) that produce the external payload at generation time on the server. External queries return only the external payload; internal views are role-gated. This avoids N per-domain receipt tables and avoids leaking internal fields through client-side hiding.

---

## 2. Rejected Alternatives

| Alternative | Rationale for rejection |
|---|---|
| **Command-journal-only snapshots** | The command journal `beforeSnapshot`/`afterSnapshot` exists for reversal/audit, not document rendering. It stores full row dumps without structured document semantics, version stability, or external projection contracts. Mixing audit snapshots with customer/vendor-facing receipts couples unrelated lifecycles and makes schema evolution fragile. |
| **Per-domain receipt tables** | `purchase_order_receipts`, `sales_order_receipts`, `payment_receipts`, etc. would duplicate timestamp/versioning/status/projection machinery and fragment queries. A shared table with a `document_type` discriminator keeps the security model, indexing, and archive policy in one place. |

---

## 3. Operator Jobs-to-be-Done

1. **PO vendor confirmation / finalization receipt** — After drafting a PO, the operator finalizes it, reviews a vendor-facing receipt. The external projection includes vendor-agreed fields (`unitCost` mapped to vendor unit price, `costRangeLow`/`costRangeHigh` mapped to vendor price range) and excludes PO internal fields (PO-level `internalNotes`, line-level internal notes, `buyerNotes`, `unitPrice`, and any field not explicitly allowlisted). For this roadmap `unitPrice` is treated as planned resale/markup, while vendor price is represented by `unitCost` or `costRangeLow`/`costRangeHigh`. The receipt snapshot is persisted and retrievable.
2. **Sales customer receipt / order confirmation without cost/margin leaks** — After confirming a sales order, the operator generates a customer-facing confirmation. The external projection excludes `unitCost`, `internalMargin`, `costRangeLow`/`costRangeHigh`, `internalNotes`, and all cost-basis/margin fields. No internal pricing or margin data may reach the customer-facing payload.
3. **Later: customer payment receipt** — When a payment is posted, generate a customer-facing payment receipt from the same snapshot foundation.
4. **Later: vendor payout receipt** — When a vendor payment is recorded, generate a vendor-facing payout receipt from the same snapshot foundation.

## 4. Proposed Data / Security Model

### 4.1 `document_snapshots` table (high-level shape)

| Column | Purpose |
|---|---|
| `id` | UUID primary key |
| `document_type` | `'purchase_order' \| 'sales_order' \| 'customer_payment' \| 'vendor_payout'` |
| `subject_id` | UUID of the source entity (PO id, SO id, payment id, etc.) |
| `version` | Integer, starting at 1, incrementing on amendment/regeneration |
| `status` | `'draft' \| 'finalized' \| 'superseded' \| 'void'` |
| `internal_payload` | JSONB — complete document shape, including all internal fields |
| `external_payload` | JSONB — generated by server-side projection, never written by client |
| `projection_version` | Integer — bumps when the per-type allowlist changes, enabling cache invalidation |
| `generated_by_command_id` | UUID → `command_journal.id` — provenance |
| `created_at` / `updated_at` | Timestamps |

Indexes: `(document_type, subject_id)`, `(subject_id, version DESC)`, `(status, document_type)`, unique `(document_type, subject_id, version)`, partial unique `(document_type, subject_id) WHERE status IN ('draft', 'finalized')`.

### 4.2 Per-type pure projection modules

Each `document_type` has a server-side module exporting:

- `EXTERNAL_FIELDS: readonly string[]` — additive allowlist of keys from `internal_payload` that may appear in `external_payload`.
- `PROJECTION_VERSION: number` — current schema version of the allowlist.
- `projectExternal(internal: unknown): { payload: unknown; projectionVersion: number }` — pure function that constructs the external payload.

No key outside `EXTERNAL_FIELDS` may reach `external_payload`. If a required key is missing from `internal_payload`, the projection fails explicitly rather than silently omitting it.

### 4.3 External projection rules

- **Server-side only.** The client never receives `internal_payload` on external endpoints. The client never decides which fields to hide via CSS, column visibility, or conditional rendering.
- **Persisted at generation time.** `external_payload` is computed and stored when the snapshot is created or amended, not recomputed on every read.
- **Projection version immutability.** Finalized snapshots are immutable. A stored `external_payload` is not automatically regenerated when the projection version bumps. Regeneration requires explicit operator action or migration. In the Tranche 1 normal refinalize path, the prior finalized row is voided (via unfinalize → `void`) before a new finalized version is created. `superseded` is a schema-reserved status for future direct regeneration/amendment paths that do not go through an intermediate unfinalize; no Tranche 1 normal path depends on `superseded`.
- **External query/text endpoints** return only `external_payload` and a plain-text render derived from it.
- **Internal view** is gated by role (`owner`, `manager`, `operator`). Viewers receive external payload only.
- **Internal copy/print** must include a literal `INTERNAL — DO NOT SEND` watermark or prefix in the rendered output.

### 4.4 Document lifecycle

Canonical Tranche 1 lifecycle (the only paths the implementation must honor):

```
            saveDraft (allowed only when parent PO.status = 'draft')
            ┌──────────────────────────────────────────────┐
            ▼                                              │
        ┌───────┐  finalize (UPDATE same row)   ┌─────────────┐  unfinalize   ┌──────┐
  ─────►│ draft │ ─────────────────────────────►│  finalized  │ ─────────────►│ void │
        └───────┘                                └─────────────┘                └──────┘
            │                                          ▲
            │ abandon (UPDATE same row)                │ finalize (INSERT v_next when no draft exists)
            ▼                                          │
         ┌──────┐                                     start
         │ void │
         └──────┘
```

1. **Save draft** is allowed only when the parent PO is in `draft` status. Save draft creates a new snapshot row at `status='draft', version = max(version) + 1` (or `version = 1` if none exists) — unless an active `draft` snapshot already exists for this subject, in which case the existing row is **updated in place** (`internal_payload`/`external_payload` refreshed, version unchanged, id unchanged).
2. **Finalize** consumes any active draft for this subject by UPDATING the same row from `status='draft'` to `status='finalized'`. The version on that row is unchanged. If no active draft exists, finalize INSERTS a new row at `status='finalized', version = max(version) + 1` (or `1` if none exists).
3. **Unfinalize** transitions the single active `finalized` row for this subject to `status='void'`.
4. **Refinalize after unfinalize** creates a new row at `status='finalized', version = max(version) + 1`. The prior row remains `status='void'`, **not** `superseded`. `superseded` is **not** the normal Tranche 1 refinalize disposition.
5. **Abandon draft** transitions the active `draft` row to `status='void'`.
6. The partial unique index `(document_type, subject_id) WHERE status IN ('draft', 'finalized')` continues to hold: draft and finalized never coexist for the same subject because finalize either updates the draft in place or inserts only after the prior finalized row was voided by unfinalize.

Status meanings:

- `draft`: Operator-editable workspace state. In Tranche 1, only allowed when the parent PO is in `draft` status; the server-side `saveDraftPurchaseOrderReceipt` handler enforces this.
- `finalized`: Frozen; external payload is generated and persisted. There is at most one finalized row per subject at any time (partial unique index).
- `void`: Explicitly invalidated. Set by abandon draft (from `draft`) or by unfinalize (from `finalized`). Voided rows remain in the table as audit history.
- `superseded`: **Schema-reserved for future direct regeneration/amendment** that does not go through an intermediate unfinalize (Phase 4+). No Tranche 1 normal path produces a `superseded` row. The status check constraint includes it so Phase 4+ migrations do not need to alter the constraint.

### 4.5 Security defaults (Tranche 1)

- **Viewer/external retrieval returns FINALIZED snapshots only.** Draft snapshots are never reachable by viewers through any external endpoint or text endpoint.
- **Operator-only draft preview path.** Internal roles (`owner | manager | operator`) MAY preview the active draft snapshot for a subject through operator-gated endpoints (`getInternalBySubjectId`, and `getReceiptText` with the operator-only `includeDrafts: true` flag). This path renders both the internal payload and the operator-fetched `external_payload` of the active draft. The viewer-facing external endpoint is unchanged and remains finalized-only.
- **External router responses are minimized.** `getExternalBySubjectId` returns only `{ version, projectionVersion, externalPayload }`; `getReceiptText` returns only `{ text, version, projectionVersion }`. No `id`, `subjectId`, `generatedByCommandId`, status, or `createdAt` is included in those responses (those fields belong to operator-only endpoints).
- **Tranche 1 router accepts only `documentType: 'purchase_order'`.** Phase 2/3 types are rejected until implemented. `getById` additionally enforces that the fetched row's `documentType === 'purchase_order'`; rows of any other type return NOT_FOUND so `getById` cannot be used as a cross-type bypass.
- **`getById` and `listVersions` are operator+ only in Tranche 1.** Viewer variants are deferred.
- **`documentSnapshots` rows are excluded from generic command-journal `snapshotByAffectedIds`** to prevent `internalPayload`/`externalPayload` from leaking into command history accessible to viewers. Receipt-related commands' `affectedIds` include the parent PO id **only**; the snapshot row id is never added to `affectedIds`. Snapshot provenance is recorded on the `documentSnapshots` row itself via `generatedByCommandId`. The exclusion is asserted by explicit tests on `queries.relatedCommands` and reversal-preview endpoints (see plan Task 10 leak-guard tests).

## 5. Phases

| Phase | Scope | Key Deliverables |
|---|---|---|
| **Phase 0 — Shared foundation** | Migration, `document_snapshots` table, base projection module contract, tRPC router (`documentSnapshots`) with internal/external endpoints, role gating, basic plain-text renderer (human-readable sentences, not Signal-optimized) | Migration SQL, schema types, router with `getById` / `getExternalBySubjectId` / `listVersions`, unit tests for projection contract |
| **Phase 1 — PO vertical** | `purchase_order` projection module, finalization workspace integration, vendor receipt preview using the external payload, copy/print with internal watermark, save draft / abandon draft | `poProjection.ts`, PO finalization workspace uses snapshot commands (`createDraftSnapshot`, `finalizeSnapshot`, `amendSnapshot`), browser/runtime proof |
| **Phase 2 — Sales vertical** | `sales_order` projection module, customer order confirmation receipt, integration with Sales workspace | `salesProjection.ts`, sales confirmation generates snapshot, external endpoint returns customer-safe payload |
| **Phase 3 — Money receipts** | `customer_payment` and `vendor_payout` projection modules, payment receipt generation | Payment and vendor payment commands emit snapshot on post |
| **Phase 4 — Hardening** | Version history UI, archive policy, accessibility review, Signal-optimized formatting refinements on top of the plain-text renderer, performance indexes | Archive older than N days, accessible print preview, Signal-optimized plaintext formatting |

## 6. First Tranche Scope

**Phase 0 (shared foundation) + Phase 1 (PO vertical) are delivered together.**

Explicitly **out of scope** for Tranche 1:
- Sales vertical (Phase 2)
- Money receipt verticals (Phase 3)
- Dedicated amendment workflow UI (re-finalization from scratch creates a new version; no separate amendment UI)
- New top-level nav lane (receipts live inside existing PO/Sales views)
- PDF generation (plain text and copy/print only)
- Signal sending integration (text export only)
- Attachments (future)
- Historical backfill of existing finalized POs

### 7.2 Tranche 1 operator-observable AC
- [ ] Finalizing a PO creates a `document_snapshots` row with `document_type = 'purchase_order'`.
- [ ] Operator can open a receipt preview from a finalized PO; preview shows vendor-safe fields only.
- [ ] Operator can open a receipt preview of an **active draft** snapshot (operator-only path); viewers cannot see drafts through any endpoint or UI control.
- [ ] Operator can copy/print the receipt; internal copy includes `INTERNAL — DO NOT SEND` watermark.
- [ ] Operator can save a draft snapshot from the finalization workspace; draft is mutable and retrievable through operator-gated endpoints only.
- [ ] Operator can abandon a draft snapshot; status transitions from `draft` → `void`.
- [ ] Unfinalizing a PO voids the active snapshot (status → `void`).
- [ ] Refinalizing after an unfinalize creates a new version (version increments). The prior finalized row remains `void` (set by the preceding unfinalize), **not** `superseded`. No Tranche 1 normal path produces a `superseded` row.
- [ ] After finalize, save draft, or abandon, the operator is returned to the purchase orders table with prior scroll position, row selection, sort, and filter state preserved.
- [ ] Copy/print text output is chat-readable plain text (sentences, line breaks, human labels) — not JSON, CSV, or structured markup.
- [ ] The Preview receipt button is visible to viewers only when an active finalized snapshot exists for the selected PO. Viewers attempting to read a draft via the viewer external endpoint receive NOT_FOUND.

### 7.3 Leak-control AC
- [ ] **PO external endpoint** returns `unitCost` (as vendor unit price), `costRangeLow`/`costRangeHigh` (as vendor price range), and other explicitly allowlisted fields. It never returns PO internal fields: PO-level `internalNotes`, line-level internal notes, `buyerNotes`, `unitPrice`, or any field not in `EXTERNAL_FIELDS`. For this roadmap `unitPrice` is treated as planned resale/markup, while vendor price is represented by `unitCost` or `costRangeLow`/`costRangeHigh`.
- [ ] **Sales external endpoint** never returns `unitCost`, `internalMargin`, `costRangeLow`/`costRangeHigh`, `internalNotes`, buyerNotes, or any cost-basis/margin field.
- [ ] **Runtime/projection invariant:** `external_payload` must contain only keys present in the registered `EXTERNAL_FIELDS` for that document type/version. Any key outside the allowlist must fail the projection and return a server error.
- [ ] **Change-control/version invariant:** any change to `EXTERNAL_FIELDS` must bump `PROJECTION_VERSION`, enforced by tests or reviewer checklist.
- [ ] No client-side code branches on `role` to decide which fields to render from an external payload.

### 7.4 Table state preservation AC
- [ ] Existing `purchase_orders` and `purchase_order_lines` columns (`internalNotes`, `externalNotes`, `finalizedAt`, `paymentTerms`, `prepaymentAmount`) remain unchanged.
- [ ] Existing `finalizePurchaseOrder`, `unfinalizePurchaseOrder`, `approvePurchaseOrder` commands continue to work; they gain snapshot side effects, not replacement logic.
- [ ] Existing PO grid and finalization UI continue to function; receipt preview is additive.

## 9. Open Questions / Blockers Before Coding

### Resolved Tranche 1 defaults (decided by this roadmap + plan)

The following items were originally open questions and are now resolved by this roadmap and the Tranche 1 plan (`docs/superpowers/plans/2026-05-20-finalization-receipts-tranche-1.md`). They are listed here for traceability; no further decision is required before Tranche 1 coding.

- **Draft persistence boundary** — **Resolved:** explicit Save Draft only. No autosave. The finalization workspace creates/updates a single active draft snapshot for the subject only when the operator clicks Save draft. The active-draft row is updated in place on subsequent saves (same id, same version).
- **Amendment behavior** — **Resolved (Tranche 1):** there is no dedicated amendment UI in Tranche 1. The supported lifecycle is `finalize → unfinalize → refinalize`. Refinalize after unfinalize creates a new version; the prior finalized row remains `void`, not `superseded`. `superseded` is schema-reserved for Phase 4+ direct-regeneration paths and is never written by Tranche 1 code.
- **Role gating granularity** — **Resolved:** `viewer` is restricted to external payloads only (and finalized snapshots only). `owner`, `manager`, and `operator` may read internal payloads and may preview active draft snapshots through operator-gated endpoints. There is no per-document-type override in Tranche 1; the read gate lives in the `documentSnapshots` router and is the server-side source of truth. The client never branches on role to mask fields inside an external payload.

### Remaining open questions

1. **Sales status model** — Sales orders currently use `draft → confirmed → posted → fulfilled → shipped → cancelled`. Does the confirmation step become the snapshot trigger, or do we add an explicit `finalized` status like PO? **Phase 2 blocker** — decision required before Phase 2 spec. Not a Tranche 1 blocker.
2. **Archive-period inclusion** — Should snapshots be included in monthly archive runs alongside their parent entities, or kept live indefinitely? Affects migration and `archiveRuns` integration. **Phase 4 concern** — does not block Tranche 1 but must be answered before archive integration ships.
