# Part 1 — The System Model

## 1.1 Personas (8, canonical)

| Persona | Lives in | Command families | Speed target / "good" |
|---|---|---|---|
| Owner / Main Manager | Dashboard, Sales, Clients, Closeout, Reports | CMD-SALES (exceptions), CMD-CLOSEOUT, CMD-PAYMENTS | Exception approval < 60s queue→confirm; dashboard actionable without scrolling |
| Sales Operator | Sales, Inventory, Orders, Clients | CMD-SALES, CMD-POSTING | New sale draft < 30s cold start; find product without leaving Sales |
| Inventory Operator | Intake, Purchase Orders, Inventory, Recovery | CMD-INTAKE, CMD-PO | Inline row creation; explicit reversible posting |
| Payments / Accounting | Payments, Vendor Payouts, Clients, Recovery | CMD-PAYMENTS, CMD-VENDOR | Log + allocate < 60s; unapplied surfaced prominently |
| Warehouse Operator | Fulfillment, Pick, Orders | CMD-FULFILLMENT | ≤2 actions per pack line; queue updates immediately |
| Support Operator | Orders, Payments, Clients, Recovery, Dashboard | read-only (+CMD-RECOVERY view) | "Where is order X?" < 60s; one-place balance explanation |
| Photographer / Readiness | Inventory, Sales, Photography | CMD-INTAKE (media) | Catalog readiness determinable < 60s; one inline edit per status |
| Connector Actor | Processors/Connectors (external submitter) | CMD-CONNECTOR | Request confirmation; visible "no ledger write yet" |

Plus the implicit ninth: **Viewer** role (read-only operator across most lanes — `accessPolicy.ts` viewer loop).

## 1.2 Work loops × primary surfaces

Buy → Purchase Orders · Receive → Intake (+Receipts) · Sell → Sales/Orders/Inventory Finder · Collect/Pay → Payments/Vendor Payouts/Client Ledger · Fulfill → Fulfillment/Pick · Recover/Close → Recovery/Closeout · Decide → Dashboard/Reports · Support → Palette/Global search/Relationship drawer/Row history/Recovery.

## 1.5 Flow inventory

- **13 documented journeys** (PO, Intake, Inventory, Sales, Fulfillment, Payments/AR, Vendor Bills/AP, Closeout, Recovery, Matchmaking, Connectors, Credit Review, Photography) — 84 branch scenarios, 52 server-side error states, 18 client gates, 15 cross-journey dependencies.
- **26 persona QA flows** (24 persona + 2 cross-persona ship gates), each normal/edge/error.
- **Key combo flows** every roadmap item below was tested against:
  1. PO → approve → draft intake → verify → post receipt → vendor bill → schedule → pay (X1, full money loop)
  2. Intake reversal mid-sale (X2 — reversal while inventory is reserved)
  3. Sale → price → confirm (credit check) → post → invoice → pick → weigh/pack → fulfill → payment FIFO → closeout marks
  4. Buyer-credit (negative payment) → later allocation → balance explanation to customer
  5. Consignment lot depletion → auto vendor-bill due → payout
  6. Matchmaking need + vendor stock → accepted match → PO or Sale creation
  7. Below-floor negotiation → reason annotation → owner review
  8. Photography: shoot session → publish media → mediaStatus gates customer sheet → catalog export
  9. Connector request → route/approve (no ledger write) → internal command
  10. Mistaken post → row history → reversal preview → reverse → re-post
  11. Period closeout: unsafe-row drilldown → fix across 4+ views → lock → archive
  12. Support reconstruction: fragment search → entity → status story → customer-safe answer
  13. Dual-role counterparty review (same entity is buyer + vendor) → directional AR/AP
  14. Quick Ledger burst: 5 mixed money rows (cash/crypto/processor) in one sitting → post all
  15. Referee-attributed sale → referee credit accrual → payout/void

## 1.6 What is already genuinely good (don't churn)

Status-aware StatusActionBar adopted on 8 surfaces with real-status decision tables; unified RowInspector (History/Relationship/Issue) on every OperatorGrid; FormDialog/InspectorDrawer convergence (A1–A4); FilterPresetStrip; finder pill-filter chrome + saved DB-backed views + natural price parsing + compare footer; Quick Ledger grid; reversal policies matrix + TER-1521 confirm-flow; closeout blocker parity + blocker routing; idempotent audited command bus; error toasts persist until dismissed; shared-workstation privacy partialize; cross-tab store sync; connection-lost banner; My Drafts; Credit Watch; per-view drawer state persistence. **The roadmap below builds on this spine — no rip-and-replace anywhere.**

# Part 2 — Scoring model

Every item: **Severity** (S3 blocks/corrupts · S2 major friction · S1 annoying · S0 polish) + **Frequency** (F3 daily · F2 weekly · F1 occasional · F0 rare) → **Priority = S+F**: **P0** (5–6, fix now) · **P1** (3–4, next sprint(s)) · **P2** (1–2, backlog) · **P3** (0, track). **Effort:** S < 1 day · M 1–3 days · L 1+ week. Item IDs are stable (`UX-§##`) for Linear import; each cites evidence (file:line or doc) and a concrete fix.

# Part 3 — The Tick List

## §A · Trust & correctness — promises the UI makes and breaks (all P0/P1)

- [ ] **UX-A01 [P0·S]** Fake "health check" affordance. `Alt+H` pushes toast "Health check requested. Watch the top status indicator." — no command runs, and there is no top status indicator. Evidence: `Hotkeys.tsx:168-171`. Fix: either wire to a real `auth.me`/server health refetch with pass/fail toast, or delete the binding. Spec §8 "no fake affordances" violation.
- [ ] **UX-A02 [P0·S]** Fake "Validate All" affordance. `Alt+V` toasts "Validate All complete: visible grids are loaded from server state." without validating anything. Evidence: `Hotkeys.tsx:172-175`. Fix: delete, or wire to a genuine refetch of the active view's grid query + `verifyAllIntake` on Intake.
- [ ] **UX-A03 [P0·M]** `⌘Enter` bypasses the status decision tables. It fires `confirmSalesOrder`/`postSalesOrder`/`allocatePayment` on **only the first selected row**, ignores multi-select, ignores row status, and contradicts the StatusActionBar primary the operator sees. Evidence: `Hotkeys.tsx:176-184`. Fix: route `⌘Enter` through the same decision table the bar uses — "commit the visible primary for the current selection" — and toast disabled-reason when no primary applies. This makes the spec's "`⌘↵` commits primary" checklist item true.
- [ ] **UX-A04 [P0·S]** CAP-024 broken: Quick Ledger drafts are NOT persisted. `ledgerDrafts` is absent from the persist allowlist, so a half-entered money session dies on reload — exactly the moment (interruption) drafts exist for. Evidence: `uiStore.ts:502-516` partialize vs registry CAP-024 "Phase 3 persists ledger drafts in UI store." Fix: persist `ledgerDrafts` (drafts contain counterparty names — same PII class as the already-persisted column prefs are not; if the shared-workstation rationale from PR #80/#89 wins, persist server-side per-user instead, but do not leave the registry promise broken). Document the decision either way.
- [ ] **UX-A05 [P0·S]** Wrong default empty-state copy on every grid. OperatorGrid's fallback reads "Create or import rows, then mark them Ready when they can be posted." — intake-specific advice rendered on Payments, Closeout, Disputes, Clients, everywhere a view didn't pass `emptyTitle`. Evidence: `OperatorGrid.tsx:662`; only ~10 views pass custom empties. Fix: neutral default ("No rows match the current view") + per-view `emptyTitle/emptyChildren` sweep with next-action guidance (see §D).
- [ ] **UX-A06 [P0·S]** Merge-candidates badge is a permanently-zero signal. `mergeCandidateCount` is queried and shown but nothing ever populates `contact_merge_candidates` (BE-014). Operators see a live-looking counter that can never fire. Evidence: `ContactsView.tsx:25`, registry BE-014/TER-1591. Fix: hide badge + MergeCandidatesView entry until the detection job ships, or ship the detection job (§U).
- [ ] **UX-A07 [P0·S]** `/` grid-filter hotkey — promised by the spec verification checklist ("`/` opens grid filter"), implemented nowhere. Evidence: no handler in `Hotkeys.tsx` or `OperatorGrid.tsx`. Fix: `/` focuses the active OperatorGrid quick-filter input (skip when editing text).
- [ ] **UX-A08 [P1·S]** Dead CSV-import machinery in IntakeView. ~50 lines of live state/handlers (`csvOpen/csvText/handleCsvDrop/importCsv`, focus trap) for the retired TER-1658 flow; backend rejects it. Evidence: `IntakeView.tsx:45-78`. Fix: delete dead code; keep the explanatory comment.
- [ ] **UX-A09 [P1·S]** Product docs contradict shipped intake policy. `work-loops.md` Loop-2 non-negotiable "Vendor receipt can be generated from selected rows without requiring a formal PO" and north-star vocabulary "Receive Inventory" both predate TER-1658 (PO-only intake). The Keel "Receive" chip still launches `receiving` quick-launch. Fix: update work-loops/north-stars to PO-first; re-point the Receive chip at "Receive against PO" (PO picker) or rename it; or, product call: restore ad-hoc receiving (BE-009 adjacent). One of the two must give.
- [ ] **UX-A10 [P1·S]** Spec §10 contains wrong status vocabularies in 4+ views (vendor bills, pick lists, connectors, command journal, payments) — already burned two implementation waves; the spec was never corrected. Evidence: decisions-log 2026-06-11 Decision 1; HANDOFF table. Fix: patch `docs/design/spec.md` §10 tables to the real values (or stamp each table "superseded — see decisions-log") so the next agent stops re-deriving.
- [ ] **UX-A11 [P1·S]** Native `confirm()` in VendorContextDrawer brand-removal — bypasses ConfirmRoot styling/focus management used everywhere else. Evidence: `VendorContextDrawer.tsx:339`. Fix: route through `useConfirm()`.
- [ ] **UX-A12 [P1·S]** Orphaned-but-reachable routes: `/processors` and `/connectors` removed from nav (TER-1664) yet still in `defaultOperatorViews` (accessPolicy.ts:5), still routed in App.tsx, still reachable via palette entity navigation. Half-dead surfaces confuse and bit-rot. Fix: decide — (a) if MVP-out, gate the routes + access policy + palette behind the same flag and redirect to Settings→Requests; (b) if staying, restore nav entries. Today it's neither.
- [ ] **UX-A13 [P1·M]** Recovery and Closeout each have two homes: top-level nav routes AND Settings tabs (`SettingsView` renders `<RecoveryView/>`/`<CloseoutView/>`/`<ConnectorsView/>` under Actions/Archive/Requests). Drawer state, selected rows, and palette deep-links key off *different* ViewKeys (`settings` vs `recovery`), so the same workflow has divergent drawer/selection behavior depending on entry path. Evidence: `OperationsViews.tsx:3105-3154`, `CommandPalette.tsx navigateEntity` (command→settings/actions). Fix: pick one home per workflow (recommend: nav routes are canonical; Settings tabs become links/redirects), and make palette deep-links target the canonical home.
- [ ] **UX-A14 [P1·S]** Capability-ID collision: "CAP-030" means *Pricing Rules Chain Manager* in the registry but *bulk release-for-picking* in the swarm handoff/decisions-log. Two teams will eventually ship the wrong thing. Fix: reconcile the registry; assign the release-for-picking lineage its own CAP row.
- [ ] **UX-A15 [P1·S]** Sheet-export partial-failure messaging: on snapshot failure the file still downloads and the pill says "Sheet downloaded, but Recent Sheets snapshot failed" — good — but the error is set via `console.error` + state with no retry affordance. Evidence: `SalesView.tsx:1122-1131`. Fix: add "Retry snapshot" action on the pill.

## §D · Feedback systems — toasts, errors, empty, loading, disabled

- [ ] **UX-D01 [P1·M]** Toasts are message-only — no actions. After posting/reversal/receipt the operator gets text with no "View order / Open receipt / Undo" deep link, forcing manual navigation to verify consequences (persona friction: "posting silently succeeds with no batch confirmation"). Evidence: `ToastCenter.tsx` (button=dismiss only); `uiStore.pushToast(message, tone)` signature. Fix: extend toast shape with optional `{actionLabel, onAction}`; adopt on the 10 highest-frequency commands (post sale, post receipt, log payment, allocate, reverse, fulfill, lock, archive, schedule, pay).
- [ ] **UX-D02 [P1·S]** Error toasts have no copy/diagnose affordance. Support packet exists, but a failing operator can't grab the command id/message in one click. Fix: error toasts include "Copy details" (command name, id, message) and, for command failures, "Open in Recovery" prefiltered.
- [ ] **UX-D03 [P1·M]** Empty-state sweep (beyond UX-A05): give every grid a next-action empty state — Payments ("No payments yet — press Money In"), Fulfillment ("No open picks — post an order to create work"), Disputes, Closeout archive, Reports (per-report), Matchmaking three grids (done), Photography ("All batches have media 🎉" vs "No batches"). Each empty state names the verb + the surface that produces rows.
- [ ] **UX-D04 [P1·S]** Disabled-without-reason sweep. StatusActionBar standardized disabled-with-reason; but 64 `disabled` usages in OperationsViews + Sale tray + finder still grey out silently (e.g., "Add suggestion" disabled with no hint that an order must be selected). Fix: every disabled action gets `title`/inline reason text; pattern: "Select an order first."
- [ ] **UX-D05 [P2·S]** Loading: AG Grid `loading` flags exist, but panel-level fetches (CustomerPurchaseHistoryPanel, drawer tabs, reports) should standardize on the `_DrawerTab` loading/empty/error wrapper; audit for spinners-with-no-text and layout jumps.
- [ ] **UX-D06 [P2·S]** Optimistic-feel latency: commands round-trip before grids refetch; high-frequency verbs (mark packed, verify intake) feel slower than a spreadsheet keystroke. Fix: per-command optimistic row patch via tRPC utils where reversal-safe (status flips), with rollback on failure.
- [ ] **UX-D07 [P2·S]** `aria-live` announcement channel exists — verify every `runCommand` result routes through it (success and failure), not just toasts, so screen-reader operators hear outcomes.
- [ ] **UX-D08 [P3·S]** Reduced-motion: drawer/panel transitions don't honor `prefers-reduced-motion`. Add the media query to the canvas-shell transitions.
